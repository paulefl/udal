== 1. Business Goals

[cols="1,3,2"]
|===
| ID | Goal | Success Criterion

| BG-01
| *Reduce IoT integration effort* — application developers integrate a new device type in < 1 day without writing protocol-specific code
| Developer survey: > 80% achieve first device integration in < 1 working day

| BG-02
| *Enable embedded deployment* — UDAL runs at the edge without a cloud dependency or an external database
| Gateway runs on Raspberry Pi 4 with ≤ 150 MB RAM at idle, zero external services required

| BG-03
| *Protocol independence* — application code must not change when a device's transport is replaced
| Replacing an MQTT device with an HTTP device requires zero changes to client application code

| BG-04
| *Open ecosystem* — third-party contributors add transport adapters without forking the core
| External adapter compiles into gateway by implementing the Transport interface; CI passes unchanged

| BG-05
| *Security by default* — every default deployment is encrypted and authenticated; insecure mode is explicit opt-in
| Security audit finds zero unauthenticated endpoints in default configuration
|===

== 3. System Scope

=== 3.1 In Scope (v1)

* UDAL Gateway: gRPC + REST API, Auth middleware, Message Router
* Transport adapters: MQTT (3.1.1 + 5), HTTP (poll + webhook), CAN (SocketCAN)
* Device Registry and Capability Registry (embedded bbolt)
* SDKs: Go (device + app), Rust (embedded), Python (scripting), TypeScript (Node.js / browser)
* Reflex Dashboard (reference web UI built with Python Reflex)
* Deployment artefacts: single binary (amd64, arm64), Docker image ≤ 30 MB, Helm chart

=== 3.2 Out of Scope (v1)

* Cloud-native SaaS management plane
* AUTOSAR Classic BSW or safety-certified embedded components
* OPC-UA, BACnet, Zigbee, Modbus adapters
* Built-in MQTT broker (external Mosquitto or compatible required)
* ABAC (Attribute-Based Access Control) — post-v1
* Device firmware OTA updates — post-v1 (see udal-automotive repo)

== 4. Functional Requirements

NOTE: Each requirement includes its acceptance criteria. Use these directly for test planning.

=== 4.1 Device Lifecycle

==== F-01 — Device Registration

[cols="1,4"]
|===
| ID | F-01
| Priority | Must-Have
| Stakeholders | SH-01, SH-02

| Description
| A device registers itself by providing an ID and a capability schema reference.
  The gateway persists the entry and validates that the schema exists before confirming registration.

| Acceptance Criteria
a|
- [ ] Device calls RegisterDevice → gateway returns success, entry visible in ListDevices
- [ ] Registration with unknown schema reference → rejected with `NOT_FOUND`
- [ ] Duplicate device ID → rejected with `ALREADY_EXISTS`
- [ ] Registered entry survives gateway restart (bbolt persistence)
|===

==== F-02 — Device Unregistration

[cols="1,4"]
|===
| ID | F-02
| Priority | Must-Have
| Stakeholders | SH-01, SH-03

| Description
| A device or admin can unregister a device. The gateway removes it from the registry
  and closes any active subscription streams for that device.

| Acceptance Criteria
a|
- [ ] UnregisterDevice → entry removed from Device Registry
- [ ] Active Subscribe streams for that device receive end-of-stream
- [ ] Subsequent GetDevice → `NOT_FOUND`
- [ ] Unregistering a non-existent device → `NOT_FOUND` (idempotent option configurable)
|===

==== F-03 — Device Discovery

[cols="1,4"]
|===
| ID | F-03
| Priority | Must-Have
| Stakeholders | SH-01

| Description
| A client lists registered devices, optionally filtered by transport type, tag, or online status.

| Acceptance Criteria
a|
- [ ] ListDevices without filter → all registered devices returned
- [ ] Filter by `transport=mqtt` → only MQTT devices returned
- [ ] Filter by `online=true` → only devices with heartbeat within timeout returned
- [ ] Filter by tag → only devices carrying that tag returned
- [ ] Filters combinable (AND semantics)
- [ ] Empty result → empty list, not error
|===

==== F-04 — Heartbeat / Online Status

[cols="1,4"]
|===
| ID | F-04
| Priority | Must-Have
| Stakeholders | SH-01, SH-03

| Description
| The gateway tracks device last-seen timestamps via heartbeat messages.
  A device that misses N consecutive heartbeats (configurable, default 3 × 30 s = 90 s)
  transitions to offline. Online status changes emit `DeviceStatusEvent` to subscribers.

| Acceptance Criteria
a|
- [ ] Device publishes heartbeat every 30 s; gateway updates `last-seen` in registry
- [ ] Device silent for > 90 s → gateway sets `online=false`, emits `DeviceStatusEvent`
- [ ] Device reconnects → gateway sets `online=true`, emits `DeviceStatusEvent`
- [ ] Heartbeat interval and timeout configurable in gateway config
|===

=== 4.2 Data Access

==== F-05 — Property Read

[cols="1,4"]
|===
| ID | F-05
| Priority | Must-Have
| Stakeholders | SH-01, SH-02

| Description
| A client reads the current value of a named device property via gRPC or REST.
  The gateway routes the request to the correct transport adapter and returns the typed value.

| Acceptance Criteria
a|
- [ ] GetProperty on MQTT device → adapter publishes request, awaits response, returns value
- [ ] GetProperty on HTTP device → adapter polls endpoint, returns value
- [ ] GetProperty on CAN device → adapter decodes last received frame for that signal
- [ ] Unknown device → `NOT_FOUND`
- [ ] Unknown property path → `NOT_FOUND`
- [ ] Adapter timeout (> 5 s default) → `DEADLINE_EXCEEDED`
- [ ] p99 latency < 50 ms under normal load (≤ 1,000 devices, ≤ 100 req/s)
|===

==== F-06 — Property Write

[cols="1,4"]
|===
| ID | F-06
| Priority | Must-Have
| Stakeholders | SH-01

| Description
| A client writes a value to a writable device property.
  The gateway validates the value type against the capability schema before forwarding.

| Acceptance Criteria
a|
- [ ] SetProperty with valid value and correct type → forwarded to adapter, device acknowledges
- [ ] SetProperty with wrong type (e.g. string for a float property) → `INVALID_ARGUMENT`
- [ ] SetProperty on read-only property → `PERMISSION_DENIED`
- [ ] SetProperty on offline device → `UNAVAILABLE`
|===

==== F-07 — Command Dispatch

[cols="1,4"]
|===
| ID | F-07
| Priority | Must-Have
| Stakeholders | SH-01

| Description
| A client sends a named command with typed parameters to a device.
  The gateway forwards it via the device's transport adapter and returns the response.

| Acceptance Criteria
a|
- [ ] SendCommand with valid name and params → forwarded to adapter, response returned
- [ ] Unknown command name → `NOT_FOUND`
- [ ] Missing required parameter → `INVALID_ARGUMENT`
- [ ] Command timeout (configurable, default 10 s) → `DEADLINE_EXCEEDED`
- [ ] Device NACK → `FAILED_PRECONDITION` with device error message in details
|===

==== F-08 — Property Subscription (Streaming)

[cols="1,4"]
|===
| ID | F-08
| Priority | Must-Have
| Stakeholders | SH-01

| Description
| A client subscribes to real-time updates of a device property via gRPC server-side streaming.
  The gateway fans out each device event to all active subscribers of that property.

| Acceptance Criteria
a|
- [ ] Client calls Subscribe → receives `PropertyEvent` each time device publishes
- [ ] Multiple clients subscribe to the same property → all receive each event (fan-out)
- [ ] Client disconnects → subscription cleaned up, no goroutine leak
- [ ] Device goes offline → subscriber receives `DeviceStatusEvent` (offline)
- [ ] Event delivery latency < 50 ms p99 from device publish to client receipt
|===

=== 4.3 Transport Adapters

==== F-09 — MQTT Adapter

[cols="1,4"]
|===
| ID | F-09
| Priority | Must-Have
| Stakeholders | SH-01, SH-02

| Description
| Supports MQTT 3.1.1 and 5. Uses request/response topic convention.
  Reconnects with exponential backoff. Circuit breaker on sustained errors.

| Topic Convention
a|
[source]
----
udal/{deviceId}/props/{path}         ← device publishes value
udal/{deviceId}/props/{path}/get     ← gateway requests value
udal/{deviceId}/props/{path}/set     ← gateway writes value
udal/{deviceId}/cmds/{name}          ← gateway sends command
udal/{deviceId}/cmds/{name}/resp     ← device responds
udal/{deviceId}/status               ← device heartbeat
----

| Acceptance Criteria
a|
- [ ] ReadProperty: gateway publishes to `.../get`, receives value within 5 s default timeout
- [ ] WriteProperty: gateway publishes to `.../set`, device confirms on `.../set/ack`
- [ ] Subscribe: fan-out to router on every incoming property event
- [ ] Reconnect: exponential backoff from 1 s to 60 s max after broker disconnect
- [ ] MQTT v5 and v3.1.1 both work (tested against Mosquitto 2.x in CI)
- [ ] Circuit breaker: 5 consecutive errors → open for 30 s, then half-open probe
|===

==== F-10 — HTTP Adapter

[cols="1,4"]
|===
| ID | F-10
| Priority | Must-Have
| Stakeholders | SH-01, SH-02

| Description
| Poll-based property reads at configurable interval. Webhook receiver for push events.
  Supports mTLS for device authentication.

| Acceptance Criteria
a|
- [ ] ReadProperty: polls device HTTP endpoint, parses JSON response to typed value
- [ ] Subscribe: webhook receiver starts per-device listener; events forwarded to router
- [ ] Poll interval configurable per device (default 5 s)
- [ ] mTLS: gateway presents client cert to device when configured
- [ ] HTTP errors (4xx/5xx) → mapped to appropriate gRPC status codes
|===

==== F-11 — CAN Adapter (SocketCAN)

[cols="1,4"]
|===
| ID | F-11
| Priority | Must-Have
| Stakeholders | SH-02

| Description
| Reads and writes raw CAN frames via SocketCAN (Linux kernel interface).
  Decodes frames to typed property values using DBC files or YAML signal maps.

| Acceptance Criteria
a|
- [ ] ReadProperty: returns decoded signal value from last received CAN frame with matching ID
- [ ] WriteProperty: encodes value to CAN frame, writes to SocketCAN interface
- [ ] DBC file loaded at startup; signal map cached in memory
- [ ] Multiplexed signals correctly decoded (MUX support)
- [ ] Tested against virtual CAN (`vcan0`) in CI
- [ ] Decode latency < 1 µs per frame (benchmark)
|===

==== F-12 — Pluggable Adapter Interface

[cols="1,4"]
|===
| ID | F-12
| Priority | Should-Have
| Stakeholders | SH-05

| Description
| Third-party adapters implement the public `Transport` interface and are wired into
  the gateway via configuration. No changes to gateway core packages required.

| Acceptance Criteria
a|
- [ ] `Transport` interface is public and documented in `gateway/internal/adapter/transport.go`
- [ ] Example third-party adapter compiles and passes the common adapter test suite
- [ ] Gateway loads adapter via config `type: custom` + binary plugin or compiled-in registration
|===

=== 4.4 Capability Schema

==== F-13 — Schema Publish

[cols="1,4"]
|===
| ID | F-13
| Priority | Must-Have
| Stakeholders | SH-02, SH-03

| Description
| An operator publishes a capability schema (YAML/JSON) to the Capability Registry.
  Schemas are immutable once published; a new version creates a new registry entry.

| Acceptance Criteria
a|
- [ ] Publish valid schema → stored in Capability Registry, retrievable by `name@version`
- [ ] Publish duplicate `name@version` → rejected with `ALREADY_EXISTS`
- [ ] Schema validated against UDAL meta-schema on publish → invalid schema rejected with `INVALID_ARGUMENT`
- [ ] Schema persists across gateway restart
|===

==== F-14 — Schema Validation on Registration

[cols="1,4"]
|===
| ID | F-14
| Priority | Must-Have
| Stakeholders | SH-02

| Description
| The gateway rejects device registration if the declared schema reference does not exist.

| Acceptance Criteria
a|
- [ ] RegisterDevice with unknown schema → `NOT_FOUND`
- [ ] RegisterDevice with valid schema → accepted
- [ ] Schema compatibility warning logged if device declares minor-version mismatch
|===

==== F-15 — Property Type Validation

[cols="1,4"]
|===
| ID | F-15
| Priority | Must-Have
| Stakeholders | SH-01

| Description
| The gateway validates SetProperty values against the declared property type in the schema.

| Acceptance Criteria
a|
- [ ] Float property receives string value → `INVALID_ARGUMENT` before forwarding to adapter
- [ ] Enum property receives value outside declared set → `INVALID_ARGUMENT`
- [ ] Value within valid range and type → forwarded to adapter
|===

=== 4.5 Security

==== F-16 — API-Key Authentication

[cols="1,4"]
|===
| ID | F-16
| Priority | Must-Have
| Stakeholders | SH-03, SH-04

| Description
| Clients include an API key in the `X-API-Key` header.
  The gateway validates against stored bcrypt hashes. Keys are never returned in API responses.

| Acceptance Criteria
a|
- [ ] Valid API key → authenticated, request proceeds
- [ ] Missing header → `UNAUTHENTICATED`
- [ ] Invalid / unknown key → `UNAUTHENTICATED`
- [ ] API keys stored as bcrypt hashes (cost ≥ 12) in Device Registry
- [ ] Key rotation: new key accepted immediately; old key rejected after rotation
|===

==== F-17 — mTLS Authentication

[cols="1,4"]
|===
| ID | F-17
| Priority | Must-Have
| Stakeholders | SH-02, SH-04

| Description
| Clients present a TLS client certificate.
  The gateway validates it against the configured CA chain. Certificate CN or SAN becomes the identity.

| Acceptance Criteria
a|
- [ ] Valid client cert from configured CA → authenticated; CN used as identity
- [ ] Missing client cert (when mTLS required) → TLS handshake fails with `CERTIFICATE_REQUIRED`
- [ ] Expired cert → `UNAUTHENTICATED`
- [ ] Cert signed by unknown CA → `UNAUTHENTICATED`
- [ ] mTLS-optional mode: requests without cert fall through to other auth methods
|===

==== F-18 — OAuth2 Device Flow (JWT Bearer)

[cols="1,4"]
|===
| ID | F-18
| Priority | Must-Have
| Stakeholders | SH-01, SH-04

| Description
| Clients include a JWT Bearer token. The gateway validates signature via configured JWKS URL
  and checks `aud`, `exp`, and `iss` claims.

| Acceptance Criteria
a|
- [ ] Valid JWT with correct `aud`/`iss`/`exp` → authenticated; `sub` claim used as identity
- [ ] Expired JWT → `UNAUTHENTICATED`
- [ ] Wrong audience → `UNAUTHENTICATED`
- [ ] Invalid signature → `UNAUTHENTICATED`
- [ ] JWKS URL unreachable → cached keys used for up to 5 min; beyond that → `UNAVAILABLE`
|===

==== F-19 — RBAC Authorisation

[cols="1,4"]
|===
| ID | F-19
| Priority | Must-Have
| Stakeholders | SH-04

| Description
| Every operation checks the caller's role against the RBAC permission matrix.

| Roles × Operations
a|
[cols="2,1,1,1,1"]
!===
! Operation ! admin ! operator ! reader ! device
! RegisterDevice ! ✅ ! ✅ ! ❌ ! ✅
! GetDevice ! ✅ ! ✅ ! ✅ ! ✅ (own)
! ListDevices ! ✅ ! ✅ ! ✅ ! ❌
! GetProperty ! ✅ ! ✅ ! ✅ ! ✅ (own)
! SetProperty ! ✅ ! ✅ ! ❌ ! ✅ (own)
! SendCommand ! ✅ ! ✅ ! ❌ ! ✅ (own)
! Subscribe ! ✅ ! ✅ ! ✅ ! ✅ (own)
!===

| Acceptance Criteria
a|
- [ ] Each role × operation combination in the matrix tested
- [ ] `reader` calls SetProperty → `PERMISSION_DENIED`
- [ ] `device` calls GetProperty on own device → permitted
- [ ] `device` calls GetProperty on foreign device → `PERMISSION_DENIED`
|===

==== F-20 — Per-Device ACL

[cols="1,4"]
|===
| ID | F-20
| Priority | Must-Have
| Stakeholders | SH-03, SH-04

| Description
| Per-device ACL entries override role-level permissions (allow or deny specific callers).

| Acceptance Criteria
a|
- [ ] ACL `allow` entry for `reader` on specific device → reader can SetProperty on that device
- [ ] ACL `deny` entry for `operator` on specific device → operator cannot access that device
- [ ] ACL checked after RBAC; ACL allow overrides RBAC deny; ACL deny overrides RBAC allow
- [ ] ACL entries persisted in Device Registry alongside device entry
|===

=== 4.6 Observability

==== F-21 — Health Endpoint

[cols="1,4"]
|===
| ID | F-21
| Priority | Must-Have
| Stakeholders | SH-03

| Description
| `GET /health` returns HTTP 200 with JSON body when the gateway is healthy and ready.

| Acceptance Criteria
a|
- [ ] Gateway ready → `GET /health` returns `200 {"status": "ok"}`
- [ ] Gateway starting (not yet ready) → `503`
- [ ] Adapter(s) failed → `200` with degraded status per adapter in body (gateway still serves other adapters)
|===

==== F-22 — Prometheus Metrics

[cols="1,4"]
|===
| ID | F-22
| Priority | Must-Have
| Stakeholders | SH-03

| Description
| `GET /metrics` exposes metrics in Prometheus exposition format.

| Required Metrics
a|
- `udal_devices_online` (gauge) — currently online devices
- `udal_requests_total{operation, status}` (counter) — total API requests
- `udal_request_duration_seconds{operation}` (histogram) — request latency
- `udal_adapter_errors_total{adapter}` (counter) — adapter-level errors

| Acceptance Criteria
a|
- [ ] All four metrics present after startup
- [ ] `udal_devices_online` increments on RegisterDevice, decrements on timeout
- [ ] `udal_request_duration_seconds` p99 bucket matches load test measurements
|===

==== F-23 — Structured Logging

[cols="1,4"]
|===
| ID | F-23
| Priority | Must-Have
| Stakeholders | SH-03, SH-04

| Description
| All log output is JSON with mandatory fields: `timestamp`, `level`, `component`, `trace_id`.
  Optional: `device_id`, `error`. Log level configurable at runtime.

| Acceptance Criteria
a|
- [ ] Every log line is valid JSON with all mandatory fields
- [ ] `UDAL_LOG_LEVEL=debug` enables debug logs without restart
- [ ] Request log line includes `trace_id` matching the OTEL span ID
|===

==== F-24 — OpenTelemetry Tracing

[cols="1,4"]
|===
| ID | F-24
| Priority | Should-Have
| Stakeholders | SH-03

| Description
| Trace context is created on every incoming request and propagated through all
  internal calls (auth → router → adapter). OTLP exporter endpoint configurable.

| Acceptance Criteria
a|
- [ ] Every gRPC request produces a trace with spans: `api`, `auth`, `router`, `adapter`
- [ ] `trace_id` is consistent across all spans in one request
- [ ] OTLP endpoint configurable via `UDAL_OTEL_ENDPOINT`; tracing disabled if unset
|===

== 6. Constraints

=== 6.1 Technical Constraints

[cols="1,3,2"]
|===
| ID | Constraint | Impact

| TC-01
| Linux ≥ 5.10 required for CAN adapter (SocketCAN)
| macOS and Windows not supported for CAN in v1; documented in README

| TC-02
| No mandatory external database (bbolt embedded store)
| Simplifies edge deployment; performance limit at > 10,000 devices (v2 concern)

| TC-03
| TLS mandatory; plain-text only via explicit `--dev` flag
| Certificates must be provisioned before first start (cert-manager or self-signed for dev)

| TC-04
| Protobuf as the API contract; all changes go through `api/proto/`
| REST clients use generated grpc-gateway proxy; no hand-written REST handlers

| TC-05
| Minimum versions: Go ≥ 1.22, Rust stable, Python ≥ 3.12, Node.js ≥ 20
| Enforced in CI; older toolchains not supported
|===

=== 6.2 Organisational Constraints

[cols="1,3,2"]
|===
| ID | Constraint | Impact

| OC-01
| Apache 2.0 licence
| All dependencies must be Apache-2.0, MIT, or BSD compatible; GPL dependencies prohibited

| OC-02
| Conventional Commits on all merged commits
| Enforced by CI; enables automatic changelog for releases

| OC-03
| All architecture changes require `architecture.jsonc` update + ADR
| See CONTRIBUTING.md §Architecture Changes

| OC-04
| No vendored dependencies
| Go modules, Cargo, pip, npm manage dependency versions; reproducible via lock files
|===

[cols="1,3"]
|===

== 7. External Interface Requirements

=== 7.1 API Interface

[cols="1,2,2"]
|===
| Interface | Protocol | Notes

| Primary API
| gRPC / protobuf over TLS
| Service definition in `api/proto/`; generated stubs in `api/proto/gen/`

| REST Gateway
| HTTPS + JSON
| Generated by grpc-gateway from proto annotations; spec in `api/openapi/`

| Streaming (Subscribe)
| gRPC server-side stream
| WebSocket bridge for TypeScript SDK and browser clients

| Metrics
| HTTP, Prometheus exposition format
| `GET /metrics` on configurable port (default 9090)

| Health
| HTTP JSON
| `GET /health` on metrics port
|===

=== 7.2 Configuration Interface

Single YAML file (`gateway.yaml`) with full override via environment variables (`UDAL_` prefix):

[source,yaml]
----
gateway:
  grpc_port: 50051          # UDAL_GRPC_PORT
  http_port: 8080           # UDAL_HTTP_PORT
  metrics_port: 9090        # UDAL_METRICS_PORT
  tls:
    cert: /etc/udal/tls/server.crt
    key:  /etc/udal/tls/server.key
    ca:   /etc/udal/tls/ca.crt
  auth:
    api_key_header: X-API-Key
    jwks_url: https://auth.example.com/.well-known/jwks.json
  registry:
    path: /var/udal/registry.db
  adapters:
    mqtt:
      broker: tcp://mosquitto:1883

=== 7.3 SDK Interface Contract

Each SDK MUST expose the following operations with language-idiomatic error handling:

[cols="2,3"]
|===
| Operation | Description

| `Connect(config)` / constructor
| Establish connection to gateway; authenticate

| `Disconnect()` / destructor
| Close connection gracefully

| `ReadProperty(deviceId, path)` → Value
| Read current property value

| `WriteProperty(deviceId, path, value)`
| Write property value

| `SendCommand(deviceId, command, params)` → Result
| Send command and await response

| `Subscribe(deviceId, path)` → Channel / AsyncIterator / Callback
| Subscribe to property updates (language-idiomatic async primitive)

| `RegisterDevice(id, schemaRef, transport)` *(device-side SDK only)*
| Register this device with the gateway
|===

Error handling per language:
* **Go**: `(Value, error)` — errors are `*udal.Error` with `Code` and `Message`
* **Rust**: `Result<Value, UdalError>`
* **Python**: raises `UdalError(code, message)`
* **TypeScript**: `Promise` — rejects with `UdalError`

== 8. Glossary

[cols="1,3"]
|===
| Term | Definition

| Adapter
| A gateway-internal plugin implementing the `Transport` interface for one protocol

| bbolt
| Embedded key/value store for Go; used for Device and Capability registries

| buf
| Protobuf toolchain for linting, formatting, and breaking-change detection

| Capability Schema
| YAML/JSON document describing a device's properties, commands, and events

| CAN
| Controller Area Network — automotive/industrial fieldbus protocol

| Circuit Breaker
| Pattern: after N consecutive errors, stop calling a failing dependency for a cooldown period

| gRPC
| Google Remote Procedure Call — binary RPC over HTTP/2 using protobuf

| grpc-gateway
| Go library that generates a REST/JSON proxy from gRPC service definitions

| MQTT
| Message Queuing Telemetry Transport — lightweight pub/sub protocol for IoT

| no_std
| Rust compilation mode excluding the standard library; suitable for bare-metal targets

| OTLP
| OpenTelemetry Protocol — standard exporter format for traces, metrics, and logs

| RBAC
| Role-Based Access Control — permissions assigned to named roles

| Reflex
| Python full-stack web framework that compiles Python to React/Next.js

| SocketCAN
| Linux kernel subsystem providing a socket-based API for CAN bus access

| Transport
| Go interface (`ReadProperty`, `WriteProperty`, `SendCommand`, `Subscribe`) implemented by each adapter
|===
