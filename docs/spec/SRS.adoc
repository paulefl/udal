== Introduction

=== Purpose

This document specifies the functional and non-functional requirements for UDAL (Universal Device Abstraction Layer), a middleware platform providing a unified API surface for heterogeneous IoT devices across multiple transport protocols.

=== Scope

UDAL covers:

* The UDAL Gateway (central runtime service)
* Transport Adapters (MQTT, HTTP, CAN)
* Device and Capability Registries
* Client SDKs (Go, Rust, Python, TypeScript)

UDAL does **not** cover:

* AUTOSAR Classic BSW or safety-certified embedded runtime
* Cloud-native SaaS management plane (post-v1)
* UI / dashboard (post-v1)

=== Definitions

[cols="1,3"]
|===
| Term | Definition

| Device
| A physical or virtual IoT endpoint that exposes properties and accepts commands

| Property
| A named, typed value on a device (e.g. `engine.rpm`, `temperature`)

| Command
| An action sent to a device (e.g. `reboot`, `calibrate`)

| Capability Schema
| A YAML/JSON document describing a device's properties, commands, and events

| Transport
| The communication protocol used between the gateway and a device (MQTT, HTTP, CAN)

| Adapter
| A gateway-internal plugin implementing the `Transport` interface for one protocol

| SDK
| A client library used by devices or applications to interact with UDAL

| RBAC
| Role-Based Access Control — permissions assigned to roles

| ACL
| Access Control List — per-device permission overrides
|===

== Overall Description

=== System Context

UDAL sits between client applications and heterogeneous IoT devices.
Client applications address devices through a single, protocol-independent API.
UDAL routes requests to the correct transport adapter and returns results.

See link:../../architecture.jsonc[architecture.jsonc] — view `context` for the system context diagram.

=== User Classes

[cols="1,3,2"]
|===
| Class | Description | Primary Interface

| Application Developer
| Builds services that consume device data or send commands
| gRPC API, REST API, SDKs (Go, Python, TypeScript)

| Device Developer
| Builds firmware or edge agents that expose device capabilities
| SDK (Go, Rust), MQTT, HTTP direct

| Platform Operator
| Deploys, configures, and monitors UDAL
| CLI, Prometheus metrics, health endpoint

| End User
| Uses applications built on top of UDAL (indirect user)
| —
|===

=== Assumptions and Dependencies

* Linux kernel ≥ 5.10 for CAN adapter (SocketCAN)
* MQTT broker (Mosquitto ≥ 2.0 or compatible) available for MQTT transport
* Go ≥ 1.22, Rust ≥ 1.75, Python ≥ 3.11, Node.js ≥ 20 for SDK development
* TLS certificates provisioned externally (e.g. cert-manager, Let's Encrypt)

== Functional Requirements

=== FR-001: Device Registration

[cols="1,4"]
|===
| ID | FR-001
| Title | Device Registration
| Description | A device SHALL be able to register itself with the gateway, providing its ID and capability schema reference. The gateway SHALL persist the device entry in the device registry.
| Priority | Must-Have
| Verification | Integration test: device registers, gateway returns success, entry visible in ListDevices
|===

=== FR-002: Property Read

[cols="1,4"]
|===
| ID | FR-002
| Title | Property Read
| Description | A client SHALL be able to read the current value of any property of a registered device via the gRPC and REST APIs. The gateway SHALL route the request to the correct transport adapter.
| Priority | Must-Have
| Verification | Integration test (all three adapters): client reads property, value matches device state
|===

=== FR-003: Property Write

[cols="1,4"]
|===
| ID | FR-003
| Title | Property Write
| Description | A client SHALL be able to write a value to a writable device property. The gateway SHALL validate the value type against the capability schema before forwarding.
| Priority | Must-Have
| Verification | Integration test: write valid value → accepted; write invalid type → rejected with structured error
|===

=== FR-004: Command Dispatch

[cols="1,4"]
|===
| ID | FR-004
| Title | Command Dispatch
| Description | A client SHALL be able to send a named command with typed parameters to a device. The gateway SHALL forward the command via the device's registered transport adapter.
| Priority | Must-Have
| Verification | Integration test: command reaches device; unknown command returns `NOT_FOUND`
|===

=== FR-005: Property Subscription (Streaming)

[cols="1,4"]
|===
| ID | FR-005
| Title | Property Subscription
| Description | A client SHALL be able to subscribe to real-time updates of a device property via gRPC server-side streaming. The gateway SHALL fan out each device event to all active subscribers.
| Priority | Must-Have
| Verification | Integration test: device publishes property update → all N subscribers receive event within 50 ms
|===

=== FR-006: Device Discovery

[cols="1,4"]
|===
| ID | FR-006
| Title | Device Discovery
| Description | A client SHALL be able to list registered devices, filtered by transport type, tag, or online status.
| Priority | Must-Have
| Verification | Unit test: filter combinations return correct subsets
|===

=== FR-007: Capability Schema Management

[cols="1,4"]
|===
| ID | FR-007
| Title | Capability Schema Management
| Description | An operator SHALL be able to publish a new capability schema version. The gateway SHALL validate that a device's declared schema exists before accepting registration.
| Priority | Must-Have
| Verification | Integration test: register device with unknown schema → rejected; publish schema → registration succeeds
|===

=== FR-008: Health and Metrics

[cols="1,4"]
|===
| ID | FR-008
| Title | Health and Metrics
| Description | The gateway SHALL expose a `/health` endpoint (HTTP 200 when healthy) and a `/metrics` endpoint (Prometheus format) including `udal_devices_online`, `udal_requests_total`, and `udal_adapter_errors_total`.
| Priority | Must-Have
| Verification | Integration test: start gateway, assert health endpoint, assert metric names present
|===

=== FR-009: Multi-Transport Support

[cols="1,4"]
|===
| ID | FR-009
| Title | Multi-Transport Support
| Description | The gateway SHALL support MQTT (3.1.1 and 5), HTTP (poll + webhook), and CAN (SocketCAN) simultaneously. A device SHALL declare its transport in its registration.
| Priority | Must-Have (v1)
| Verification | Each adapter tested independently; integration test with mixed-transport device set
|===

=== FR-010: Authentication

[cols="1,4"]
|===
| ID | FR-010
| Title | Authentication
| Description | The gateway SHALL support three authentication mechanisms: API-Key (header `X-API-Key`), mTLS (client certificate), and OAuth2 Device Flow (JWT Bearer token). Each request SHALL be authenticated before processing.
| Priority | Must-Have
| Verification | Integration test per auth method: valid credential → accepted; missing/invalid → `UNAUTHENTICATED`
|===

=== FR-011: Authorisation (RBAC + ACL)

[cols="1,4"]
|===
| ID | FR-011
| Title | Authorisation
| Description | Access SHALL be controlled by role (admin, operator, reader, device) and per-device ACL overrides. A reader role SHALL NOT be able to write properties or send commands. ACL entries SHALL override role defaults.
| Priority | Must-Have
| Verification | Unit tests: each role × operation matrix; ACL override tested explicitly
|===

== Non-Functional Requirements

=== NFR-001: Latency

[cols="1,4"]
|===
| ID | NFR-001
| Title | End-to-End Property Read Latency
| Description | The p99 latency for a Property Read (client API request → adapter call → device response → client response) SHALL be < 50 ms on standard hardware (4 vCPU, 4 GB RAM) under normal load (≤ 1,000 concurrent devices, ≤ 100 req/s).
| Measurement | Load test with k6 or custom Go benchmark; measure p99 latency
|===

=== NFR-002: Scalability

[cols="1,4"]
|===
| ID | NFR-002
| Title | Concurrent Device Support
| Description | A single gateway instance SHALL support 1,000 concurrent connected devices without degrading below NFR-001 latency targets, on a host with 4 vCPU and 4 GB RAM.
| Measurement | Load test: 1,000 devices × 10 req/s; measure resource usage and p99 latency
|===

=== NFR-003: Availability

[cols="1,4"]
|===
| ID | NFR-003
| Title | Graceful Degradation
| Description | If one transport adapter fails (e.g. MQTT broker unreachable), the gateway SHALL continue serving other adapters. The failed adapter SHALL report `UNAVAILABLE` on affected device requests and retry connection with exponential backoff.
| Measurement | Integration test: kill MQTT broker, assert CAN and HTTP adapters still respond
|===

=== NFR-004: Security

[cols="1,4"]
|===
| ID | NFR-004
| Title | TLS Everywhere
| Description | All network connections (client↔gateway, gateway↔broker, SDK↔gateway) SHALL use TLS 1.2 minimum. Plain-text connections SHALL be rejected in production mode.
| Measurement | Static config check; integration test with and without TLS
|===

=== NFR-005: Deployment Portability

[cols="1,4"]
|===
| ID | NFR-005
| Title | Deployment Targets
| Description | The gateway SHALL be distributable as: (a) a single statically-linked binary for Linux amd64 and arm64, (b) a Docker image ≤ 30 MB, (c) a Helm chart for Kubernetes.
| Measurement | CI build produces all three artefacts; Docker image size check in CI
|===

=== NFR-006: Observability

[cols="1,4"]
|===
| ID | NFR-006
| Title | Structured Logging and Tracing
| Description | The gateway SHALL emit structured JSON logs (level, timestamp, trace-id, component). OpenTelemetry trace context SHALL be propagated across internal boundaries.
| Measurement | Log format check in integration test; trace ID present in all log lines of one request
|===

=== NFR-007: Reliability

[cols="1,4"]
|===
| ID | NFR-007
| Title | No Memory Leaks under Sustained Load
| Description | After 1 hour of sustained load (500 devices, 50 req/s), gateway heap growth SHALL be < 10 MB as measured by pprof.
| Measurement | Soak test in CI (nightly); pprof heap snapshot before/after
|===

== External Interface Requirements

=== API Interface

* gRPC (protobuf): primary API — see `api/proto/`
* REST/HTTP: grpc-gateway translation — see `api/openapi/`
* WebSocket: subscribe endpoint for TypeScript SDK

=== Configuration Interface

Single YAML config file; all values overridable via environment variables (`UDAL_*` prefix).

=== Metrics Interface

Prometheus exposition format at `GET /metrics`.

=== SDK Interface

Each SDK MUST implement: `Connect`, `Disconnect`, `ReadProperty`, `WriteProperty`, `SendCommand`, `Subscribe`.
Language-idiomatic error handling (Go: error, Rust: Result, Python: exception, TS: Promise reject).
